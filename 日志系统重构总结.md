### 3️⃣ 并发竞争解决方案

#### **问题根源（已解决）：**

❌ **修复前：**
```
goroutine 1: fmt.Printf("[1/10] 开始处理\n")
goroutine 2: fmt.Printf("[2/10] 开始处理\n")  ← 输出交叉！
goroutine 3: fmt.Printf("专辑下载失败\n")
UI goroutine: \033[10A (光标上移) ← 覆盖了上面的日志！
```

✅ **修复后：**
```
所有日志 → logger.Info/Error/... 
          → LogMutex.Lock() 
          → 检测UI模式
             ├─ UI活跃? → 写入LogBuffer（缓存）
             └─ UI结束? → 立即输出
          → LogMutex.Unlock()
```

#### **关键技术措施：**

1. **全局输出锁** (`logger.LogMutex`)
   - 保护所有向 `os.Stdout` 的写入
   - 杜绝并发输出交叉混乱

2. **UI模式检测** (`isUIMode()`)
   - 检查 `core.TrackStatuses` 长度
   - UI活跃期间：日志缓存到 `LogBuffer`
   - UI结束后：调用 `FlushBuffer()` 批量输出

3. **分离UI与日志**
   - UI使用ANSI转义序列原地刷新（`\033[%dA`）
   - 日志使用普通换行符（`\n`）
   - 两者通过时间维度分离，不再冲突

## 🔍 技术细节

### 1. UI模式检测逻辑

```go
func isUIMode() bool {
    core.UiMutex.Lock()
    defer core.UiMutex.Unlock()
    return len(core.TrackStatuses) > 0
}
```

- `core.TrackStatuses` 在UI启动前初始化为空切片
- `go ui.RenderUI(doneUI)` 启动后，长度>0
- `close(doneUI)` 后，UI停止但TrackStatuses仍存在
- 因此需要在UI结束后立即调用 `FlushBuffer()`

### 2. 日志缓冲流程

```
并发goroutine输出 → logger.Error("xxx")
                   ↓
              isUIMode()?
             ┌─────┴─────┐
          Yes│          │No
             ↓           ↓
     LogBuffer.append   fmt.Print
             │           (立即输出)
             │
        (缓存)
             │
    UI结束: FlushBuffer()
             ↓
         逐条输出
```

### 3. 锁层次结构

```
应用层（各goroutine）
    │
    ├─> logger.Info/Error/...
    │   └─> LogMutex.Lock()
    │       └─> fmt.Print(...)
    │           └─> os.Stdout.Write (原子)
    │
    └─> ui.PrintUI()
        └─> UiMutex.Lock()
            └─> LogMutex.Lock()  (通过logger.SafePrint)
                └─> fmt.Print(...)
```

**注意：** `LogMutex` 是最底层锁，所有输出都必须经过它。

## 🚨 注意事项

### 1. 避免直接使用 fmt.Printf

**错误示例：**
```go
fmt.Printf("下载完成\n")  // ❌ 绕过了并发保护！
```

**正确示例：**
```go
logger.Success("下载完成")  // ✅
```

### 2. UI结束后必须刷新缓冲区

**错误示例：**
```go
wg.Wait()
close(doneUI)
// ❌ 忘记刷新，缓存的日志丢失！
return nil
```

**正确示例：**
```go
wg.Wait()
close(doneUI)
time.Sleep(200 * time.Millisecond)
ui.PrintUI()
logger.FlushBuffer()  // ✅ 刷新缓存
logger.SectionEnd()
return nil
```

### 3. core/state.go 保持使用 fmt

因为 `logger` 依赖 `core`（检查 `core.Debug_mode`），如果 `core` 也依赖 `logger`，会形成循环依赖。

**解决方案：**
- `core/state.go` 的配置加载日志保持使用 `fmt`
- 这些日志在程序启动阶段输出，不涉及并发

## 🔧 后续修复与优化（2025-10-08晚）

### 修复1：UI重复日志问题
**问题：** Track 1 of 10 重复输出15次
**根因：** UI渲染goroutine未完全退出就进行了下一次UI操作
**解决方案：**
```go
// 添加uiWg等待UI goroutine完全退出
var uiWg sync.WaitGroup
uiWg.Add(1)
go func() {
    defer uiWg.Done()
    ui.RenderUI(doneUI)
}()

wg.Wait()
close(doneUI)
uiWg.Wait() // 确保UI goroutine完全退出

// 打印最终状态
core.UiMutex.Lock()
if len(core.TrackStatuses) > 0 {
    ui.PrintUI()
}
core.UiMutex.Unlock()

logger.FlushBuffer()
```
