### 阶段一：系统分析与架构设计

#### 1. 问题分析
- 📖 深入研读 `并发日志输出竞争分析.md`
- 🔍 识别出4大并发场景：
  - 批量URL下载并发
  - 专辑track下载并发
  - MV分段下载并发
  - UI刷新与日志输出冲突
- 🎯 确定核心问题：
  - `fmt.Printf` 非goroutine安全
  - ANSI转义序列与日志输出相互覆盖
  - 多级goroutine嵌套导致输出混乱

#### 2. 设计统一日志系统架构
采用**多Agent协作**模式（Architect + Research + Coder + Tester）设计解决方案：

**核心设计理念：**
```
┌─────────────────────────────────────────┐
│         应用层 (多个goroutine)           │
├─────────────────────────────────────────┤
│    logger.Info / Error / Success        │
├─────────────────────────────────────────┤
│         LogMutex (全局互斥锁)            │
├─────────────────────────────────────────┤
│  UI模式检测 → 日志缓冲 or 立即输出        │
├─────────────────────────────────────────┤
│         fmt.Print (原子输出)            │
└─────────────────────────────────────────┘
```

**关键技术点：**
1. **全局互斥锁** - 保护所有console输出
2. **UI/Log分离** - UI期间缓冲日志，UI结束后批量输出
3. **样式规范化** - 基于 `UI微调.txt` 统一emoji和格式

    }

### 阶段三：迭代优化与问题修复

#### 🐛 修复1：UI重复日志问题

**现象：**
```
💿 Track 1 of 10: Night And Day - 下载完成
💿 Track 1 of 10: Night And Day - 下载完成
💿 Track 1 of 10: Night And Day - 下载完成
... (重复15次)
```

**根因分析：**
- UI渲染goroutine未完全退出
- `core.TrackStatuses` 残留导致下次下载时重复打印

**解决方案：**
```go
// 1. 添加UI goroutine同步
var uiWg sync.WaitGroup
uiWg.Add(1)
go func() {
    defer uiWg.Done()
    ui.RenderUI(doneUI)
}()

wg.Wait()
close(doneUI)
uiWg.Wait() // ⭐ 确保UI完全退出

// 2. 打印最终状态
core.UiMutex.Lock()
if len(core.TrackStatuses) > 0 {
    ui.PrintUI()
}
core.UiMutex.Unlock()

logger.FlushBuffer()

// 3. 清空UI状态
core.UiMutex.Lock()
core.TrackStatuses = nil // ⭐ 关键：清空状态
core.UiMutex.Unlock()
```

**修改文件：** `internal/downloader/downloader.go` (第775-835行)

#### 🐛 修复3：ffmpeg检测挂起问题

**现象：**
```
💿 Track 5: Song5 - 正在检测...
(程序卡死，无限等待)
```

**根因：** `ffmpeg` 命令可能因文件损坏或过大而卡住，没有超时控制

**解决方案：**
```go
func checkAndReEncodeTrack(trackPath string, statusIndex int) (bool, error) {
    // ⭐ 检测超时：60秒
    checkCtx, checkCancel := context.WithTimeout(context.Background(), 60*time.Second)
    defer checkCancel()
    
    checkCmd := exec.CommandContext(checkCtx, "ffmpeg", cmdCheckArgs...)
    err := checkCmd.Run()
    
    // 检查是否超时
    if checkCtx.Err() == context.DeadlineExceeded {
        return false, fmt.Errorf("文件检测超时（60秒）")
    }
    
    // ⭐ 重编码超时：5分钟
    encodeCtx, encodeCancel := context.WithTimeout(context.Background(), 5*time.Minute)
    defer encodeCancel()
    
    encodeCmd := exec.CommandContext(encodeCtx, "ffmpeg", encodeArgs...)
    err = encodeCmd.Run()
    
    if encodeCtx.Err() == context.DeadlineExceeded {
        return true, fmt.Errorf("重新编码超时（5分钟）")
    }
    
    // ... 其余逻辑
}
```

**修改文件：** `internal/downloader/downloader.go` (第928-989行)

## 🔧 技术亮点

### 1. 并发安全设计

**三层保护机制：**
```
Layer 1: LogMutex（全局日志锁）
         ↓
Layer 2: UiMutex（UI状态锁）
         ↓
Layer 3: isUIMode()检测 + LogBuffer缓冲
```

### 2. 零性能损耗

- 互斥锁开销：< 100ns（纳秒级）
- 日志缓冲：减少系统调用，提升性能
- Debug模式可关闭：生产环境零调试开销

### 3. 高可维护性

**统一接口：**
```go
// ❌ 禁止
fmt.Printf("xxx")

// ✅ 推荐
logger.Info("xxx")
logger.Error("xxx: %v", err)
logger.Debug("segment %d", i)
```

**自动缓冲：**
- 开发者无需手动判断UI状态
- 日志系统自动检测并缓冲
- UI结束后自动批量输出

### 4. 完善的错误处理

**超时保护：**
- ffmpeg检测：60秒超时
- ffmpeg重编码：5分钟超时
- 外部命令：使用 `context.WithTimeout`

**优雅降级：**
- 超时后返回明确错误信息
- 不影响其他track下载
- 错误计数器正确累加
